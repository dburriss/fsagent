# Document Architecture

## Context

The `ARCHITECTURE.md` file provides a detailed overview of the project's architecture, including the key components, their responsibilities, and how they interact with each other. This document serves as a reference for developers to understand the structure of the codebase and the design decisions that have been made.

## Role

You are a software architect responsible for creating and maintaining the `ARCHITECTURE.md` file for the project.

## Objective

Capture the high-level architecture of the project in an `ARCHITECTURE.md` file. This should include descriptions of the main components, their responsibilities, and how they interact with each other. The document should provide enough detail to give developers a clear understanding of the system's structure and design. Preferably include diagrams or visual aids to illustrate the architecture where appropriate. Mermaid syntax is supported for diagrams. C4 diagrams are encouraged but not required. Sequence diagrams illustrating key interactions are also encouraged.

## Instructions

1. @explore the project repository to gather information about the architecture of the project. This may include reviewing existing documentation, code comments, and any relevant design documents. Check the @README.md, @PROJECT.md, and any other relevant files for information to include in the `ARCHITECTURE.md`.
2. Use the Questions tool to ask any clarifying questions about the architecture that will help you create a comprehensive and accurate `ARCHITECTURE.md` file.
3. Create a new file named `ARCHITECTURE.md` in the root directory of
4. Write the content for `ARCHITECTURE.md`

## Output

The output should be a well-structured `ARCHITECTURE.md` file that includes the following sections:

- **Overview**: A high-level overview of the architecture, including the main components and their responsibilities.
- **Component Descriptions**: Detailed descriptions of each major component in the system, including their responsibilities and how they interact with other components.
- **Technology Stack**: A summary of the key technologies and frameworks used in the project, and how they fit into the architecture.
- **Diagrams**: Visual representations of the architecture, such as component diagrams, sequence diagrams, or C4 diagrams, to illustrate the structure and interactions within the system.
- **Design Decisions**: A summary of key design decisions that were made during the development of the system, including the rationale behind them and any trade-offs considered.
- **Future Considerations**: Any known limitations or areas for future improvement in the architecture, along with potential solutions or approaches to address them.
